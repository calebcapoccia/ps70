<div class="textcontainer typewriter-texture"> <p class="margin"></p> <h3>Final Project: Kachow Car Display</h3> <p>My final project is a car-mounted scrolling LED display controlled over Bluetooth from a mobile app with speech-to-text input. A driver (or passenger) can dictate a message into the app, have it transcribed via on-device STT, and send it via BLE to the ESP32 in the display, which scrolls the message across a MAX7219 LED matrix affixed to the rear dashboard.</p> <img src="./integrated_headline.png" alt="Kachow car display headline" style="max-width: 600px; display:block; margin:0 auto;"> <h4>1. MVP</h4> <p>During MVP week, I focused on getting the core input/output pipeline working end-to-end: speech-to-text input in a browser, transmitting the transcribed message to the ESP32 over WiFi via an HTTP server, and scrolling the received text across the MAX7219 display. I also built a first-pass housing with 3D-printed clamps to hold the LED matrix to a rigid backing, and wrote the Arduino logic for non-blocking scrolling with configurable speed, pause, and repeat count.</p> <p>This got the idea functional, but relying on a laptop-hosted web server on the same WiFi network as both the phone and the display, plus finicky browser STT, made the UX brittle. For the full MVP writeup, see the <a href="../07_outputs/index.html">Week 7 page</a>.</p> <div style="width:100%; max-width:800px; margin: 20px auto;"> <div style="position: relative; padding-bottom: 56.25%; height: 0; overflow: hidden;"> <iframe style="position: absolute; top: 0; left: 0; width: 100%; height: 100%;" src="https://www.youtube.com/embed/VisRwon42es" frameborder="0" allow="accelerometer; autoplay; clipboard-write; encrypted-media; gyroscope; picture-in-picture" allowfullscreen> </iframe> </div> </div> <h4>2. Networking</h4> <p>For the networking milestone, I replaced the WiFi + web-server setup with Bluetooth Low Energy (BLE) and a native React Native iOS app, which eliminates the need for any laptop or shared WiFi. The ESP32 advertises a <code>CarDisplay</code> BLE service; the app scans, connects, and writes JSON payloads (text, speed, repeat, pause) to a characteristic, which the Arduino parses with <code>ArduinoJson</code>. I used native iOS speech recognition via <code>@react-native-voice/voice</code> for reliable on-device STT, and added a momentary-switch button on the ESP32 that force-disconnects paired clients when held for 5 seconds. For the full networking writeup, see the <a href="../09_networking/index.html">Week 9 page</a>.</p> <div style="width:100%; max-width:800px; margin: 20px auto;"> <div style="position: relative; padding-bottom: 56.25%; height: 0; overflow: hidden;"> <iframe style="position: absolute; top: 0; left: 0; width: 100%; height: 100%;" src="https://www.youtube.com/embed/UDNO3XOB1e8" frameborder="0" allow="accelerometer; autoplay; clipboard-write; encrypted-media; gyroscope; picture-in-picture" allowfullscreen> </iframe> </div> </div> <h4>3. Integration</h4> <p>Given that I already had a primitive housing from MVP week, the goal for this stretch was to finalize most of the "guts" of the project, which includes all of the visible input and output of the device. This includes:</p> <ul> <li>Incorporating the battery</li> <li>Incorporating the status LED</li> <li>USB-C port</li> </ul> <p>This will put me in a good place to work on finalizing the housing, the mechanism for affixing the device to the interior of a car, and fine-tuning the user experience.</p> <h5>Battery</h5> <p>The battery subsystem needed to:</p> <ul> <li>Power the ESP32 and the LED matrix from a single rechargeable cell</li> <li>Charge over a USB-C port</li> <li>Expose a measurable voltage for a state-of-charge estimate</li> <li>Surface that estimate in the mobile app</li> </ul> <p>I first tried using the ESP32 Lipo board with a boost converter that would convert the 3.3V output to 5V for the MAX7219, but after some testing realized the MAX7219 was pulling too much current from the board and it was causing brownouts.</p> <img src="./integrated_1.png" alt="Brownout testing with ESP32 Lipo" style="max-width: 400px; display:block; margin:0 auto;"> <p>Consequently, I decided to switch back to a normal ESP32 Dev board and use a different external Lipo battery. I switched to using a 1R0 charging module, which had a built-in boost converter, so by implementing a USB-C input I had purchased on Amazon the wiring would look like USB-C &rarr; Charging Module (with connection to Lipo) &rarr; 5V &rarr; LED display.</p> <p>I then set up a voltage divider from the battery to roughly measure the battery percentage:</p> <pre><code><span class="comment">// analogReadMilliVolts uses ESP32's factory ADC calibration</span> <span class="keyword">uint32_t</span> mv = <span class="function">analogReadMilliVolts</span>(batteryPin); <span class="keyword">if</span> (pct &lt; <span class="number">0</span>) pct = <span class="number">0</span>; <span class="keyword">if</span> (pct &gt; <span class="number">100</span>) pct = <span class="number">100</span>; </code></pre> <p>Next, I incorporated an RGB LED as an indicator for the battery (green for 80%+, yellow for 21%&ndash;79%, and red for 20% or below) with appropriate resistors (220&Omega; for red and 330&Omega; for the green and blue pins).</p> <p>After putting together the circuitry, it seemed to work, but without soldering it looked like one big mess.</p> <img src="./integrated_2.jpg" alt="Breadboard mess before soldering" style="max-width: 400px; display:block; margin:0 auto;"> <p>I decided to use a small ElectroCookie breadboard, and to fit in it, an ESP32 Mini.</p> <img src="./integrated_3.jpg" alt="ESP32 Mini on ElectroCookie breadboard" style="max-width: 400px; display:block; margin:0 auto;"> <p>This is the current circuitry (which I will update and make a lot prettier for the final version).</p> <img src="./integrated_4.jpeg" alt="Current circuitry diagram" style="max-width: 400px; display:block; margin:0 auto;"> <p>However, I think I removed one of the copper rings, because the panel stopped working. Using a multimeter, I determined it was the VCC pin, so fortunately I was able to solder to another VCC on the chain and it started working again. Here's the complete soldered guts!</p> <img src="./integrated_7.jpg" alt="Complete soldered internals" style="max-width: 400px; display:block; margin:0 auto;"> <p>For now, I am keeping the same housing until I finalize the housing, but the key interaction pieces (USB-C, button, light) are all able to rest just outside the box.</p> <img src="./integrated_8.jpg" alt="Housing with USB-C, button, and LED exposed" style="max-width: 400px; display:block; margin:0 auto;"> <p>Also, for the yellow light, it is not mixing the red and green, so I will need to figure out a way to disperse the light.</p> <img src="./integrated_9.jpg" alt="Yellow LED mixing issue" style="max-width: 400px; display:block; margin:0 auto;"> <p>Finally I updated the Arduino code to publish the battery state over a BLE characteristic:</p> <pre><code><span class="comment">// Publish over BLE</span> <span class="keyword">if</span> (pBatteryCharacteristic != <span class="keyword">nullptr</span>) { <span class="keyword">char</span> buf[<span class="number">48</span>]; <span class="keyword">int</span> intPct = (<span class="keyword">int</span>)(pct + <span class="number">0.5f</span>); <span class="function">snprintf</span>(buf, <span class="keyword">sizeof</span>(buf), <span class="string">"{\"v\":%.2f,\"p\":%d}"</span>, voltage, intPct); lastReportedPct = intPct; } } </code></pre> <p>And then subscribed to it from the React Native app to display in the UI:</p> <pre><code><span class="comment">// Initial battery read</span> <span class="keyword">try</span> { <span class="keyword">const</span> initial = <span class="keyword">await</span> discovered.<span class="function">readCharacteristicForService</span>( SERVICE_UUID, BATTERY_CHARACTERISTIC_UUID, ); <span class="keyword">try</span> { <span class="keyword">const</span> parsed = JSON.<span class="function">parse</span>(<span class="function">fromBase64</span>(char.value)); <span class="function">setBattery</span>({ voltage: <span class="type">Number</span>(parsed.v), percent: <span class="type">Number</span>(parsed.p) }); Final Stretch</h4> <p>Other end-of-the-semester projects ended up taking a lot of my time entering reading week. I then had roughly one week left to make improvements to the project. There were a number of things I could work on, like a published mobile app or a PCB-milled circuit board. However, while these would be nice add ons, the top priorities were to modify the setup to drain less power and to finalize housing. As in the past, I used generative AI tools to accelerate development for some aspects of the code and documentation.</p> <h5>Sleep Mode</h5> <p>Given that the ESP32 is constantly running and the MAX7219 is always draining power, I wanted a way of "turning off" the device without disconnecting the ESP32 from the breadboard, such that I would not need to recharge the battery every time I want to use the device again.</p> <p>I could accomplish this through setting the ESP32 to deep sleep mode, which would take significantly less battery, from the order of perhaps hours on the 3400mAh battery to months when in sleep mode. I added this code to put the device to sleep:</p> <pre><code><span class="keyword">void</span> <span class="function">enterDeepSleep</span>() { Serial.<span class="function">println</span>(<span class="string">"Entering deep sleep..."</span>); ledDisplay.<span class="function">clear</span>(); <span class="keyword">if</span> (deviceConnected) { pServer-&gt;<span class="function">disconnect</span>(connectionId); <span class="function">delay</span>(<span class="number">100</span>); } <span class="comment">// Wait for button to be released before sleeping</span> <span class="keyword">while</span> (<span class="function">digitalRead</span>(btnPin) == LOW) { <span class="function">delay</span>(<span class="number">10</span>); } <span class="function">delay</span>(<span class="number">50</span>); <span class="comment">// Extra debounce</span> <span class="comment">// Ensure pull-up is enabled and held during sleep</span> <span class="function">rtc_gpio_pullup_en</span>((gpio_num_t)btnPin); <span class="function">rtc_gpio_pulldown_dis</span>((gpio_num_t)btnPin); <span class="comment">// Use EXT1 wake source (supports more pins, wake on LOW)</span> <span class="function">esp_sleep_enable_ext1_wakeup</span>(<span class="number">1ULL</span> &lt;&lt; btnPin, ESP_EXT1_WAKEUP_ALL_LOW); <span class="function">esp_deep_sleep_start</span>(); } </code></pre> <p>However, I would also need to wake it up with the button, so I soldered the button to an RTC pin. I still had the issue with the LED not mixing colors, so I decided to experiment with and then switch to a NeoPixel LED.</p> <img src="./final_progress_photos/final_progress_1.jpg" alt="Button soldered to RTC pin" style="max-width: 400px; display:block; margin:0 auto;"> <p>Even in deep sleep, the MAX7219s and the NeoPixel LED would still draw current, so I would need some way of logically gating the power to the lights. I decided to add a P-channel MOSFET because it allowed me to cleanly switch to the 5V rail while keeping all of the grounds connected.</p> <div style="display: flex; flex-wrap: wrap; justify-content: center; gap: 20px; margin: 20px 0;"> <img src="./final_progress_photos/final_progress_2.png" alt="MOSFET circuit diagram" style="height: 300px; width: auto;"> <img src="./final_progress_photos/final_progress_3.jpg" alt="MOSFET implementation" style="height: 300px; width: auto;"> </div> <p>Additionally, I found a 1x8 MAX7219 display, so integrated this instead of the 2 1x4 displays I had prior, which were bending at their soldered connection.</p> <img src="./final_progress_photos/final_progress_4.jpg" alt="1x8 MAX7219 display" style="max-width: 400px; display:block; margin:0 auto;"> <p>I also swapped out the button to a nicer one that I could integrate into the housing, as well as changing the USB-C port to one that I could embed in the housing wall as well.</p> <div style="display: flex; flex-wrap: wrap; justify-content: center; gap: 20px; margin: 20px 0;"> <img src="./final_progress_photos/final_progress_5.jpg" alt="New button" style="height: 250px; width: auto;"> <img src="./final_progress_photos/final_progress_6.jpg" alt="New USB-C port" style="height: 250px; width: auto;"> </div> <p>For the new circuitry with the MOSFET, while breadboarding I started experiencing brownout issues, likely from the MAX7219 draining too much power when drawing from 5V through the MOSFET. Consequently, I added a 1000uF capacitor connected to the MOSFET drain and ground to absorb the current spikes and prevent the flickering brownouts. Additionally, I added a 10K ohm resistor between the gate and the source to prevent the MOSFET from randomly partially turning on.</p> <img src="./final_progress_photos/final_progress_7.jpg" alt="Capacitor and resistor added" style="max-width: 400px; display:block; margin:0 auto;"> <p>This is the final circuit diagram for the setup.</p> <img src="./final_progress_photos/final_progress_8.jpg" alt="Final circuit diagram" style="max-width: 600px; display:block; margin:0 auto;"> <h6>A Few Limitations</h6> <p>Given that the USB-C port does not contain the circuitry required for a C to C connection, I currently need to use USB-A to USB-C cables to power the device. Additionally, when the device is "off" in deep sleep, the charging module seems to shut off due to lack of current draw, and it only boots up again when plugging in the USB-C connection.</p> <h5>Final Housing</h5> <p>While I liked the MVP aesthetics of wood, I knew that I wanted to 3D print a sleek housing for the LEDs and all of the components inside, especially so that I could mount it inside of a car.</p> <h6>Outer Housing</h6> <p>For the MVP box, the lid comes off, but because I want to mount this device from the top or bottom, the entire box would need to be securely locked. To seal the box from the outside, I designed a slot mechanism that is fastened by a secured nut on the inside.</p> <div style="display: flex; flex-wrap: wrap; justify-content: center; gap: 20px; margin: 20px 0;"> <img src="./final_progress_photos/final_progress_9.jpg" alt="Slot mechanism design" style="height: 300px; width: auto;"> <img src="./final_progress_photos/final_progress_10.jpg" alt="Slot mechanism closeup" style="height: 300px; width: auto;"> </div> <p>For testing out this mechanism, along with several other pieces of the housing, I conducted mini test prints where I could make sure the prints would align with the other components in the system (like a button) before doing large prints that would eat up printer time.
